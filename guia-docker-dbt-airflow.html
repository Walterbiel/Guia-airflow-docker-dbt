01

Docker

↗ Docs oficiais

Docker empacota sua aplicação, dependências e configurações em um container — uma unidade isolada e reproduzível que roda igual em qualquer máquina. Chega de "funciona no meu computador".

Sem Docker
Cada desenvolvedor instala dependências manualmente. Versões diferentes causam bugs difíceis de rastrear. Deploy é imprevisível.
Com Docker
Todo o ambiente vive em um arquivo. Qualquer pessoa clona o repositório e roda docker compose up. O resultado é sempre idêntico.
ℹ️
Dois arquivos principais: Dockerfile (define como construir a imagem) e docker-compose.yml (define como rodar múltiplos containers juntos).
Dockerfile — Template Base arquivo template

O Dockerfile é uma receita que diz ao Docker como montar sua imagem. Cada linha é uma instrução executada em sequência durante o docker build. → Referência completa

📄 Dockerfile dockerfile 
FROM python:3.11-slim          # Imagem base — Python 3.11 versão leve (~50MB)

WORKDIR /app                     # Define o diretório de trabalho dentro do container

COPY requirements.txt .          # Copia só o requirements.txt ANTES do restante do código

RUN pip install --no-cache-dir \  # Instala dependências sem salvar cache (mantém a imagem menor)
    -r requirements.txt

COPY . .                          # Copia todo o código da máquina local para /app

EXPOSE 8000                      # Documenta que o container usa a porta 8000 (não abre sozinho)

CMD ["python", "app.py"]         # Comando padrão ao iniciar o container
FROM python:3.11-slim A imagem base é o ponto de partida. slim é uma variante mais leve que omite ferramentas desnecessárias. Alternativas: python:3.11-alpine (menor ainda) ou python:3.11 (completo). Ver todas no Docker Hub
WORKDIR /app Cria e acessa o diretório /app dentro do container. Todos os comandos subsequentes rodam a partir daqui. Evita trabalhar no diretório raiz /.
COPY requirements antes do código Técnica de otimização de cache: o Docker reutiliza camadas que não mudaram. Se você só alterou código (não dependências), o pip install não roda novamente — o build fica muito mais rápido.
CMD vs ENTRYPOINT CMD define o comando padrão mas pode ser sobrescrito em tempo de execução. ENTRYPOINT é fixo. Para scripts principais, geralmente CMD basta. Saiba mais
docker-compose.yml — Template Base arquivo template

O Compose orquestra múltiplos containers como um serviço único. Com um só comando (docker compose up) você sobe a aplicação, banco de dados, cache — tudo junto e conectado. → Referência do Compose

📄 docker-compose.yml yaml 
version: "3.9"              # Versão do schema do Compose (3.x é a atual)

services:

  app:                         # Nome do serviço (você escolhe)
    build: .                  # Usa o Dockerfile na pasta atual para construir a imagem
    container_name: meu_app  # Nome explícito do container (opcional)
    ports:
      - "8000:8000"           # HOST:CONTAINER — porta local:porta do container
    volumes:
      - .:/app               # Monta a pasta local dentro do container (hot reload)
    environment:
      - ENV=dev              # Variáveis de ambiente injetadas no container
    depends_on:
      - db                   # Garante que 'db' inicia antes de 'app'

  db:
    image: postgres:15        # Usa imagem pronta (sem Dockerfile próprio)
    container_name: meu_postgres
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: usuario    # Variáveis específicas da imagem postgres
      POSTGRES_PASSWORD: senha
      POSTGRES_DB: banco
    volumes:
      - postgres_data:/var/lib/postgresql/data  # Volume nomeado: persiste dados entre reinicios

volumes:
  postgres_data:             # Declara o volume nomeado para o Compose gerenciar
ports "8000:8000" Formato PORTA_HOST:PORTA_CONTAINER. Se a porta 8000 do host estiver ocupada, mude para "8080:8000" — o container continua usando 8000, mas você acessa em localhost:8080.
volumes ".:/app" Bind mount: sincroniza em tempo real a pasta local com o container. Edite o código e veja as mudanças sem rebuildar a imagem. Ideal para desenvolvimento, não para produção.
depends_on ⚠️ Garante apenas que o container inicia antes, não que o serviço está pronto. Para esperar o Postgres aceitar conexões, use healthcheck ou ferramentas como wait-for-it.
volumes nomeados Diferente do bind mount, volumes nomeados são gerenciados pelo Docker. Os dados do Postgres sobrevivem a docker compose down. Para apagar os dados também, use docker compose down -v.
💡
Comandos essenciais: docker compose up -d (sobe em background), docker compose logs -f app (acompanha logs), docker compose down (para e remove containers).
Exemplo completo — App Python + PostgreSQL exemplo

Um setup real com aplicação Python se conectando ao banco via variáveis de ambiente.

📄 docker-compose.yml — App + Postgres yaml 
version: "3.9"

services:
  app:
    build: .
    container_name: app_python
    ports:
      - "8000:8000"
    volumes:
      - .:/app
    depends_on:
      - postgres
    environment:             # App lê estas variáveis com os.environ.get()
      - DB_HOST=postgres      # Nome do serviço vira hostname na rede interna do Compose
      - DB_PORT=5432
      - DB_NAME=meubanco
      - DB_USER=walter
      - DB_PASSWORD=123456

  postgres:
    image: postgres:15
    container_name: postgres_db
    environment:
      POSTGRES_DB: meubanco
      POSTGRES_USER: walter
      POSTGRES_PASSWORD: 123456
    ports:
      - "5432:5432"           # Expõe para ferramentas externas (DBeaver, TablePlus...)
⚠️
Em produção, nunca coloque senhas diretamente no docker-compose.yml. Use um arquivo .env na raiz e referencie com ${DB_PASSWORD}. Adicione .env ao .gitignore.
02

dbt

↗ Docs oficiais

dbt (data build tool) traz as boas práticas de engenharia de software para as transformações SQL: modularidade com ref(), testes automatizados, documentação gerada e lineage visual. Você escreve SQL, o dbt cuida do resto.

O que o dbt faz
Compila e executa modelos SQL no seu data warehouse (BigQuery, Snowflake, Redshift, Postgres...). Gerencia dependências entre modelos e materializações (view, table, incremental).
O que o dbt NÃO faz
dbt não move dados (isso é o EL do ELT). Ele só faz o T (Transform). Para ingestão use Airbyte, Fivetran, ou custom pipelines.
📁
Estrutura de pastas padrão: models/ (SQLs), seeds/ (CSVs estáticos), tests/ (testes custom), macros/ (funções reutilizáveis), dbt_project.yml e profiles.yml.
dbt_project.yml — Configuração do projeto arquivo template

O arquivo central de configuração do dbt. Define nome do projeto, caminhos dos arquivos e configurações globais dos modelos (como materialização padrão por camada). → Referência completa

📄 dbt_project.yml yaml 
name: 'meu_projeto_dbt'        # Nome do projeto (usado como namespace)
version: '1.0.0'
config-version: 2              # Versão do schema de configuração (sempre 2)

profile: 'meu_profile'          # Referencia o perfil em profiles.yml

model-paths: ["models"]         # Onde ficam os modelos SQL
seed-paths: ["seeds"]           # Onde ficam os CSVs de seed
test-paths: ["tests"]           # Onde ficam os testes customizados

models:
  meu_projeto_dbt:              # Deve bater com o campo 'name' acima

    staging:                    # Camada de staging: limpeza e padronização
      +materialized: view      # View é rápido de criar, sem custo de storage

    marts:                      # Camada final: pronta para consumo (BI, analistas)
      +materialized: table     # Table é mais rápida para queries de BI
MaterializaçãoQuando usarCusto
viewDados que mudam frequentemente, stagingSem storage extra, query mais lenta
tableMarts finais, queries frequentes de BIStorage + tempo de build
incrementalTabelas grandes, processa só dados novosMais complexo, muito mais rápido
ephemeralSubqueries intermediárias, sem materializarZero storage, lógica inline
profiles.yml — Conexão com o banco arquivo template

Fica em ~/.dbt/profiles.yml (fora do repositório por segurança). Define como o dbt se conecta ao data warehouse em cada ambiente. → Ver todos os adapters

📄 ~/.dbt/profiles.yml yaml 
meu_profile:                   # Deve bater com 'profile' no dbt_project.yml
  target: dev                  # Qual output usar por padrão
  outputs:

    dev:                         # Ambiente de desenvolvimento
      type: postgres            # Adapter: postgres, bigquery, snowflake, redshift...
      host: localhost
      user: usuario
      password: senha
      port: 5432
      dbname: meu_banco
      schema: analytics          # Schema onde os modelos serão criados
      threads: 4                  # Número de modelos rodando em paralelo

    prod:                        # Ambiente de produção (separado)
      type: postgres
      host: prod-db.empresa.com
      schema: analytics_prod
      threads: 8
⚠️
Nunca comite o profiles.yml. Ele contém credenciais. Em CI/CD, use variáveis de ambiente: host: "{{ env_var('DBT_HOST') }}".
Modelos SQL — source(), ref() e camadas template

O coração do dbt. Cada modelo é um arquivo .sql com uma query SELECT. O dbt compila e executa no banco. Use source() para dados brutos e ref() para referenciar outros modelos.

📄 models/staging/stg_clientes.sql sql + jinja 
-- source() referencia tabelas brutas definidas em sources.yml
-- Argumento 1: nome da source, Argumento 2: nome da tabela
select
    id         as cliente_id,   -- Renomeia para padronizar nomenclatura
    nome,
    email,
    created_at as data_criacao
from {{ source('origem', 'clientes') }}
where email is not null         -- Filtros de qualidade ficam no staging
📄 models/intermediate/int_pedidos_por_cliente.sql sql + jinja 
-- ref() cria uma dependência explícita entre modelos
-- dbt resolve a ordem de execução automaticamente pelo DAG
select
    cliente_id,
    count(*) as qtd_pedidos,
    sum(valor_total) as valor_total,
    max(data_pedido) as ultimo_pedido
from {{ ref('stg_pedidos') }}      -- Referencia o modelo de staging
group by cliente_id
📄 models/marts/dim_clientes_completo.sql sql + jinja 
-- Mart: modelo final com join de múltiplas fontes
-- Este é o modelo que o BI vai consumir
select
    p.pedido_id,
    p.cliente_id,
    c.nome      as nome_cliente,
    c.email,
    p.valor_total,
    p.data_pedido
from {{ ref('stg_pedidos') }} p
left join {{ ref('stg_clientes') }} c
    on p.cliente_id = c.cliente_id
💡
Arquitetura em camadas: staging (dados brutos limpos) → intermediate (lógica de negócio) → marts (tabelas prontas para consumo). Cada camada referencia somente a anterior com ref(). Ver guia de estrutura
03

Airflow

↗ Docs oficiais

Apache Airflow é a plataforma de orquestração mais usada em engenharia de dados. Você define pipelines como código Python (DAGs), agenda execuções, monitora falhas e gerencia dependências entre tarefas com uma interface visual completa.

DAG
Directed Acyclic Graph — a definição do pipeline. Contém tarefas e as dependências entre elas. Um DAG é um arquivo Python na pasta dags/.
Operator
Template de tarefa. PythonOperator roda Python, BashOperator roda shell, PostgresOperator roda SQL. Centenas de operadores disponíveis via providers.
DAG — Template Base template

Todo DAG segue esta estrutura: imports, default_args, definição do DAG com contexto (with DAG(...)), e tarefas com suas dependências. → Conceitos de DAGs

📄 dags/dag_exemplo.py python 
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime, timedelta

# default_args são aplicados a TODAS as tarefas do DAG
default_args = {
    "owner": "data_team",          # Responsável (visível na UI)
    "retries": 1,                  # Tentativas em caso de falha
    "retry_delay": timedelta(minutes=5)  # Tempo de espera entre tentativas
}

def minha_tarefa():
    print("Executando tarefa")

# 'with DAG(...)' é o context manager que registra as tarefas no DAG
with DAG(
    dag_id="dag_exemplo",          # Identificador único na UI do Airflow
    default_args=default_args,
    schedule="@daily",            # Cron ou preset: @hourly, @weekly, None
    start_date=datetime(2026,3,1), # Data de início para backfill
    catchup=False                  # False = não executa runs passadas
) as dag:

    tarefa = PythonOperator(
        task_id="executar_funcao",   # ID único DENTRO do DAG
        python_callable=minha_tarefa # Função Python a executar
    )
schedule="@daily" Presets disponíveis: @once, @hourly, @daily, @weekly, @monthly, None (só manual). Ou use cron direto: "0 6 * * *" (todo dia às 6h). Valide seu cron aqui
catchup=False Com True, o Airflow executaria TODAS as runs desde start_date até hoje. Para novos DAGs, sempre use False para evitar dezenas de execuções indesejadas.
retries e retry_delay Em pipelines de produção, configure pelo menos 2–3 retries com delay progressivo. Falhas de rede transitórias resolvem sozinhas. Use email_on_failure=True nos default_args para alertas.
Pipeline ETL com dependências exemplo

O operador >> (bitshift) define a ordem de execução. A tarefa à direita só começa quando a esquerda conclui com sucesso.

📄 dags/pipeline_etl.py python 
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

# Funções separadas do DAG — facilita teste unitário
def extrair():
    print("Extraindo dados da API...")

def transformar():
    print("Aplicando transformações...")

def carregar():
    print("Carregando no data warehouse...")

with DAG(
    dag_id="pipeline_etl",
    schedule="@daily",
    start_date=datetime(2026, 3, 1),
    catchup=False
) as dag:

    t_extrair = PythonOperator(task_id="extrair", python_callable=extrair)
    t_transf  = PythonOperator(task_id="transformar", python_callable=transformar)
    t_carregar= PythonOperator(task_id="carregar", python_callable=carregar)

    # Define a cadeia de execução: extrair → transformar → carregar
    t_extrair >> t_transf >> t_carregar

    # Para branches paralelas: [t_a, t_b] >> t_c (a e b rodam juntas, c espera ambas)
Chamando dbt dentro do Airflow exemplo

A integração mais comum: o Airflow orquestra quando o dbt roda. Use BashOperator para um setup simples, ou o provider oficial airflow-provider-dbt para controle granular por modelo.

📄 dags/pipeline_com_dbt.py python 
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.operators.bash import BashOperator
from datetime import datetime

def ingerir_dados():
    print("Movendo dados brutos para o warehouse...")

with DAG(
    dag_id="pipeline_elt_completo",
    schedule="0 5 * * *",          # Todo dia às 5h da manhã
    start_date=datetime(2026, 3, 1),
    catchup=False
) as dag:

    # 1) Ingestão: move dados brutos para o DW
    ingestao = PythonOperator(
        task_id="ingestao",
        python_callable=ingerir_dados
    )

    # 2) dbt run: executa todos os modelos
    dbt_run = BashOperator(
        task_id="dbt_run",
        bash_command="cd /opt/airflow/dbt && dbt run --profiles-dir ."
        # Alternativa para rodar só um modelo: dbt run --select stg_clientes
    )

    # 3) dbt test: valida qualidade dos dados após transformação
    dbt_test = BashOperator(
        task_id="dbt_test",
        bash_command="cd /opt/airflow/dbt && dbt test --profiles-dir ."
    )

    # Fluxo: ingestão → dbt run → dbt test
    ingestao >> dbt_run >> dbt_test
💡
Dica pro: Instale o astronomer-cosmos para transformar cada modelo dbt em uma tarefa separada no Airflow — você ganha visibilidade e re-execução granular por modelo.
04

Conectando tudo

A stack completa rodando junta com um único docker compose up: Postgres como data warehouse, Airflow como orquestrador, e o projeto dbt montado como volume.

docker-compose completo — Airflow + dbt + Postgres exemplo
📄 docker-compose.yml — Stack completa yaml 
version: "3.9"

x-airflow-common: # Âncora YAML — evita repetição de config
  &airflow-common
  image: apache/airflow:2.9.0
  environment:
    AIRFLOW__CORE__EXECUTOR: LocalExecutor
    AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: postgresql+psycopg2://airflow:airflow@postgres/airflow
  volumes:
    - ./dags:/opt/airflow/dags        # Seus DAGs
    - ./dbt:/opt/airflow/dbt          # Projeto dbt
  depends_on:
    - postgres

services:

  postgres:                           # Banco para metadados do Airflow E seus dados
    image: postgres:15
    environment:
      POSTGRES_USER: airflow
      POSTGRES_PASSWORD: airflow
      POSTGRES_DB: airflow
    volumes:
      - postgres_data:/var/lib/postgresql/data

  airflow-webserver:                  # Interface web do Airflow (localhost:8080)
    <<: *airflow-common               # Herda a âncora definida acima
    command: webserver
    ports:
      - "8080:8080"

  airflow-scheduler:                  # Scheduler: monitora e dispara as runs
    <<: *airflow-common
    command: scheduler

volumes:
  postgres_data:
🚀
Para subir: docker compose up airflow-init (inicializa o banco do Airflow), depois docker compose up -d. Acesse a UI em http://localhost:8080 com usuário airflow / senha airflow.
